## `themeInstalls` Drizzle schema

Added in phase 4 only. Phase 2-3 have no `themeInstalls` table.

```typescript
// lib/db/schema.ts (phase 4 addition)
export const themeInstalls = pgTable('theme_installs', {
  id: text('id').primaryKey().$defaultFn(() => createId()),
  userId: text('user_id').notNull().references(() => user.id, { onDelete: 'cascade' }),
  themeId: text('theme_id').notNull(),            // matches manifest.id
  version: text('version').notNull(),              // semver from manifest
  source: text('source').notNull(),                // 'marketplace' | 'plugin' | 'custom-upload'
  manifestJson: text('manifest_json').notNull(),   // raw JSON for re-compilation
  compiledCss: text('compiled_css').notNull(),      // output of compileManifest()
  installedAt: timestamp('installed_at').defaultNow().notNull(),
  updatedAt: timestamp('updated_at').defaultNow().notNull(),
}, (table) => [
  uniqueIndex('theme_installs_user_theme').on(table.userId, table.themeId),
]);
```

Key constraints:
- `(userId, themeId)` is unique â€” one install per theme per user, updated in place on version bumps
- `source` tracks provenance: marketplace install, plugin dependency, or manual upload
- `compiledCss` is stored so it's served without recompilation on every request
- `manifestJson` is kept for re-compilation if the compile function changes

## Install flow

### From the marketplace browser

1. User browses themes in `app/(app)/settings/marketplace/page.tsx` (Themes tab)
2. User clicks "Install" on a theme card
3. Frontend calls `POST /api/themes/install { themeId, version }`
4. Backend:
   a. Fetches `downloadUrl` from the cached `index.json`
   b. Downloads the manifest JSON
   c. Validates against `ThemeManifestSchema`
   d. Runs safety checks (see below)
   e. Compiles to CSS via `compileManifest()`
   f. Inserts into `themeInstalls`
   g. Returns `{ installId, themeId, name }`
5. Frontend shows the new theme in the pack picker

### From a plugin install

1. Plugin manifest declares `"theme": { ... }` (inline JSON manifest)
2. Plugin install handler validates and compiles the theme alongside the plugin
3. Inserts into `themeInstalls` with `source = 'plugin'`
4. Theme appears in pack picker with "via plugin" badge

### Custom upload

1. User navigates to `app/(app)/settings/appearance/custom/page.tsx`
2. Pastes raw JSON manifest into a text area
3. Frontend calls `POST /api/themes/install { manifest, source: 'custom-upload' }`
4. Same validation + compile + insert flow, but no `downloadUrl` fetch

## Safety checks (`lib/themes/safety.ts`)

Run on every install, regardless of source:

1. **Schema validation** â€” must pass `ThemeManifestSchema.parse()`
2. **Size limit** â€” raw manifest JSON must be under 64KB
3. **CSS injection scan** â€” compiled CSS is scanned for:
   - `@import` rules pointing outside the font allowlist
   - `url()` values pointing outside the font allowlist
   - `expression()`, `behavior:`, `-moz-binding` (IE/Firefox CSS injection vectors)
   - `javascript:` protocol in any value
4. **Font allowlist** â€” `fontImports` URLs must match one of:
   - `fonts.googleapis.com`
   - `rsms.me`
   - `cdn.jsdelivr.net`
   - (extensible via env var `KAIROS_FONT_ALLOWLIST`)
5. **Token completeness** â€” all 20 required tokens must be present; missing tokens emit a warning but don't block install (fallback to default pack values)
6. **ID uniqueness** â€” `themeId` must not collide with a built-in pack ID

## Serving compiled CSS

`app/api/themes/[installId]/css/route.ts`:

```typescript
export async function GET(req: Request, { params }: { params: { installId: string } }) {
  const install = await db.query.themeInstalls.findFirst({
    where: eq(themeInstalls.id, params.installId),
  });
  if (!install) return new Response('Not found', { status: 404 });

  return new Response(install.compiledCss, {
    headers: {
      'Content-Type': 'text/css',
      'Cache-Control': `public, max-age=31536000, immutable`,
      'ETag': `"${install.themeId}@${install.version}"`,
    },
  });
}
```

Cache is aggressive (1 year, immutable) because the URL includes the `installId` â€” version bumps create a new row with a new ID, so the old URL naturally expires.

## Submission process

Community contributors submit themes via PR to `kairos-themes-registry`:

1. Fork the registry repo
2. Add a manifest JSON file to `manifests/`
3. Open a PR
4. CI validates the manifest automatically (`scripts/validate.ts`)
5. A maintainer reviews the preview tokens visually (CI renders a preview card)
6. Merge regenerates `index.json`
7. Theme appears in the marketplace within minutes (apps re-fetch the index on a 1-hour TTL)

No signing in v1 of the marketplace. The review step is manual. If spam becomes a problem, add a simple approval flag to `index.json` entries.

## Update flow

When a theme author pushes a new version:

1. They update the manifest in `manifests/` with a bumped `version` field
2. PR + CI + merge (same as initial submission)
3. Users who have the theme installed see an "Update available" badge in the pack picker
4. Clicking "Update" re-fetches the manifest, re-validates, re-compiles, and updates the `themeInstalls` row in place
5. The old `installId` is preserved â€” the CSS route serves the new compiled output

## In-app marketplace browser

`app/(app)/settings/marketplace/page.tsx` â€” shared UI with the plugin marketplace:

- Two tabs: **Plugins** | **Themes**
- Themes tab shows a grid of cards, each rendered from `preview` tokens (canvas bg, surface border, fg text, accent dot)
- Search by name, filter by `colorScheme` (light/dark/system)
- "Install" button on each card
- "Installed" badge + "Update" button if a newer version exists in the registry
- "Remove" button for installed themes (reverts `activeThemeId` to default if active)

## What does NOT ship in phase 4

- Premium/paid themes â€” no revenue split, no Stripe integration for themes
- Theme ratings or reviews â€” just install counts (anonymous, hosted-only)
- Automatic updates â€” user must explicitly click "Update"
- Theme versioning beyond semver â€” no "channels" or "beta" tracks
- Theme compilation service â€” compilation happens client-side (in the Vercel function), not in a separate service
