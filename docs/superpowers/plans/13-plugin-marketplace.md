# 13 — Plugin Marketplace

Distribution, install flow, safety checks, signing, and registry infrastructure for community plugins. Read `05-plugin-system.md` for the architectural overview and the plugin contract, and `references/theme-marketplace.md` for the sibling spec that this one mirrors.

Ships as **phase 4b** (see `12-phase-4b-completion.md`), sharing infrastructure with the existing theme marketplace.

**Distribution model (ADR-R15):** hybrid. Built-in plugins bundled at build time; community plugins run over HTTP.

---

## Registry: `public/plugin-registry/`

Flat-file registry mirroring the theme registry. Lives in the main Kairos repo at `public/plugin-registry/` rather than as a separate `kairos-plugins-registry` repo, because:

- One fewer repo to maintain for phase 4b
- CI already runs in the main repo; no need to duplicate the validator in a second repo
- Registry is served from `/plugin-registry/index.json` on the same origin as the app (no CORS setup)
- Migration path to a separate repo later is trivial if the submission volume ever justifies it

Structure:

```
public/plugin-registry/
├── index.json              <- generated index, regenerated on every merge
├── manifests/
│   ├── instagram.json
│   ├── twitter.json
│   ├── readwise.json
│   ├── voice.json
│   ├── gmail.json
│   └── ...
└── README.md               <- submission instructions
```

```json
      "capabilities": ["llm", "http-fetch"],
      "provenance": "https://raw.githubusercontent.com/kairos-app/kairos-plugin-instagram/main/.sigstore/bundle.json",

    },
    {
      "id": "text-to-tasks-plus",

      "author": "community-contributor",
      "version": "2.1.0",
      "distribution": "bundled",

```

`index.json` is regenerated by CI (`.github/workflows/validate-plugin-registry.yml`) on every merge touching `public/plugin-registry/manifests/`. The app fetches this index to populate the marketplace with a 1-hour TTL cache.

### Manifest shape (`PluginManifestSchema` in `@kairos/plugin-sdk`)

```typescript
import { z } from 'zod';
import { ThemeManifestSchema } from './theme-manifest';

export const PluginManifestSchema = z.object({
  schemaVersion: z.literal(1),
  id: z.string().regex(/^[a-z0-9][a-z0-9-]*$/),
  name: z.string().min(1).max(64),
  description: z.string().min(10).max(500),
  author: z.string(),
  version: z.string().regex(/^\d+\.\d+\.\d+$/),

  // Distribution
  distribution: z.enum(['http', 'bundled']),
  endpoint: z.string().url().optional(),         // required if distribution === 'http'
  npmPackage: z.string().optional(),             // required if distribution === 'bundled'

  // Behaviour
  handlesInputTypes: z.array(z.enum(['text', 'url', 'share', 'voice', 'file'])).min(1),
  capabilities: z.array(z.enum(['llm', 'http-fetch', 'file-read'])).default([]),

  // Config schema — Zod-serialised JSON, rendered as a settings form by the UI
  configSchema: z.record(z.unknown()).optional(),
  defaultConfig: z.record(z.unknown()).optional(),

  // Optional theme shipped with the plugin
  theme: ThemeManifestSchema.optional(),

  // Provenance (optional in v1, may become required in v2)
  provenance: z.string().url().optional(),

  // Presentation
  icon: z.string().url().optional(),
  homepage: z.string().url().optional(),
  repository: z.string().url().optional(),

  // Changelog — markdown string, shown in the update dialog
  changelog: z.string().optional(),
}).refine(
  (m) => m.distribution !== 'http' || !!m.endpoint,
  { message: 'HTTP plugins must provide an endpoint URL' }
).refine(
  (m) => m.distribution !== 'bundled' || !!m.npmPackage,
  { message: 'Bundled plugins must provide an npmPackage name' }
);
```

## `pluginInstalls` schema changes

The existing table (already in `lib/db/schema/plugins.ts`) needs two new columns added via a new migration:

```typescript
pluginInstalls {
  // existing columns (keep)
  id: text (primary key)
  userId: text (FK)
  pluginName: text
  pluginVersion: text
  source: text                          // 'builtin' | 'marketplace' | 'custom-upload' | 'plugin'
  sourceUrl: text | null
  enabled: boolean (default true)
  installedAt: timestamp

  // new in phase 4b
  manifestJson: jsonb                    // raw manifest, for re-verification on read
  previousVersion: text | null           // for single-step rollback
  previousManifestJson: jsonb | null
  endpoint: text | null                  // cached endpoint URL for HTTP plugins
  lastHealthyAt: timestamp | null        // HTTP plugin health tracking
}
```

Unique index on `(userId, pluginName)` — one install per plugin per user. New version overwrites in place, with `previousVersion` preserving rollback capability.

Migration: `drizzle/0004_plugin_install_fields.sql`.

## Install flow

### From the marketplace browser

1. User browses plugins in `app/(app)/settings/marketplace/page.tsx` (Plugins tab)
2. Clicks "Install" on a plugin card
3. Frontend calls `POST /api/plugins/install { pluginId, version }`
4. Backend (`lib/plugins/install.ts`):
   1. Fetches `manifestUrl` from the cached `index.json`
   2. Validates against `PluginManifestSchema`
   3. Runs safety checks (see below)
   4. If manifest has `theme`, calls `installManifest()` (theme install) with `source: 'plugin'`
   5. For HTTP plugins: `GET manifest.endpoint/manifest` — must match the registry manifest hash
   6. For bundled plugins: plugin is already in the bundle; install just records the row
   7. Inserts/upserts into `pluginInstalls`, populates `manifestJson`
   8. Calls `plugin.onInstall?(context)`
   9. Returns `{ installId, pluginName, version, theme?: { installId } }`
5. Frontend shows the plugin in the Plugins list and (if theme shipped) in the pack picker

### Custom upload

1. User navigates to `app/(app)/settings/plugins/custom/page.tsx` (new page, mirrors `appearance/custom`)
2. Pastes JSON manifest into a text area, or enters a raw manifest URL
3. Frontend calls `POST /api/plugins/install { manifest | manifestUrl, source: 'custom-upload' }`
4. Same validation + safety + insert flow, no registry `index.json` lookup required
5. **HTTP plugins via custom upload bypass the registry's implicit review.** The safety layer still runs — URL allowlist, manifest size limit, CSS injection scan on any shipped theme — but the user is told explicitly: "You're installing an unreviewed plugin. It will have access to your scratchpad content."

### Plugin updates

1. `useInstalledPlugins()` includes a computed `updateAvailable: boolean` and `availableVersion: string | null`
2. UI shows an "Update available" badge on the plugin card
3. Click "Update" opens a dialog showing the changelog from the new manifest
4. Confirm → same install flow, but with the existing row; `previousVersion` + `previousManifestJson` are populated from the current values before overwrite
5. Config, memory, and rulesets are preserved automatically (they live in `scratchpadPluginConfigs`, keyed by `(userId, pluginName)`, not by install ID)

### Rollback

1. User clicks "Rollback" (visible only if `previousVersion` is non-null)
2. Swap: `version` ↔ `previousVersion`, `manifestJson` ↔ `previousManifestJson`
3. Config/memory/rulesets stay untouched
4. If the previous version is incompatible with current config, the plugin's `onInstall` hook is responsible for migrating forward, not backward — accept data loss on rollback across breaking config schema changes

## Safety checks (`lib/plugins/safety.ts`)

Run on every install regardless of source:

1. **Schema validation** — `PluginManifestSchema.parse()` must pass
2. **Size limit** — raw manifest JSON must be under 64KB
3. **URL allowlist (HTTP plugins)** — `endpoint` must use `https://` and must not resolve to private IP ranges (10.x, 172.16-31.x, 192.168.x, 127.x, 169.254.x). DNS rebinding defence: resolve once at install, store the resolved IP, check again on every call.
4. **npm package validation (bundled plugins)** — `npmPackage` must be a valid npm name and, if `provenance` is present, must resolve to a version with matching Sigstore attestation
5. **Provenance verification (if present)** — verify the Sigstore bundle using `sigstore` npm package; on failure, refuse install
6. **Capabilities review** — any capability in `capabilities` that requires user consent (`file-read`, `http-fetch`) must be surfaced in the install dialog
7. **Shipped theme** — if `manifest.theme` is present, run the full theme safety check on it before the plugin install proceeds
8. **ID uniqueness** — `id` must not collide with a built-in plugin ID (`text-to-tasks`)
9. **Handler declaration** — `handlesInputTypes` must not be empty; must contain only known input types

## HTTP plugin runtime

### Contract

HTTP plugins expose two endpoints under a base URL (`manifest.endpoint`):

```
GET  /manifest     → PluginManifest (must match registry manifest byte-for-byte up to timestamp fields)
POST /parse        → body: { input: ScratchpadInput, config, memory, rulesets }, returns ParseResult
```

### Request signing

All calls from Kairos to an HTTP plugin are HMAC-signed with a per-(user, plugin) secret, generated on install and stored in `pluginInstalls.endpointSecret` (new field). Plugin authors verify the HMAC before processing. Prevents:

- Arbitrary callers hitting a plugin endpoint and consuming the author's compute quota
- Replay attacks (the signature covers a `timestamp` header with a 30-second tolerance)

The secret is rotated on uninstall-then-reinstall.

### Timeout, retry, circuit breaker

- Timeout: 5 seconds per parse call (plugins should do async ingestion themselves if they need longer)
- Retry: 0 in v1 (LLM calls are non-idempotent, retrying blindly could charge the user twice)
- Circuit breaker: after 3 failures in 1 minute, plugin is marked `unhealthy`; UI shows the plugin as red; `lastHealthyAt` tracks last success. After 5 minutes, allow one probe request to re-open the breaker.

### Adapter implementation

```typescript
// lib/plugins/http-adapter.ts
export function createHttpAdapter(
  install: PluginInstall,
): ScratchpadPlugin {
  return {
    name: install.pluginName,
    version: install.pluginVersion,
    displayName: install.manifestJson.name,
    description: install.manifestJson.description,
    author: install.manifestJson.author,
    handlesInputTypes: install.manifestJson.handlesInputTypes,

    canHandle(input) {
      return install.manifestJson.handlesInputTypes.includes(input.inputType);
    },

    async parse(input, context) {
      const config = await context.getConfig();
      const memory = await context.getMemory();
      const rulesets = await context.getRulesets();

      const body = JSON.stringify({ input, config, memory, rulesets });
      const timestamp = Date.now();
      const signature = await hmac(install.endpointSecret, `${timestamp}.${body}`);

      const res = await fetch(`${install.endpoint}/parse`, {
        method: 'POST',
        headers: {
          'content-type': 'application/json',
          'x-kairos-timestamp': String(timestamp),
          'x-kairos-signature': signature,
        },
        body,
        signal: AbortSignal.timeout(5000),
      });

      if (!res.ok) throw new PluginHttpError(res.status, await res.text());
      return ParseResultSchema.parse(await res.json());
    },
  };
}
```

The host treats this adapter exactly like a bundled plugin — it satisfies the `ScratchpadPlugin` interface, nothing else calls it directly.

## Signing (optional in v1, may become required in v2)

### Why optional

Making signing required in v1 creates a chicken-and-egg problem: the initial 10 plugins have to ship with provenance before users can install anything. Sam would have to author+sign all of them himself, which is both a bottleneck and defeats the point of community plugins.

### How it works (when opted in)

Plugin authors publishing to npm from GitHub Actions can add one flag:

```yaml
# .github/workflows/publish.yml
- run: npm publish --provenance --access public
  env:
    NODE_AUTH_TOKEN: ${{ secrets.NPM_TOKEN }}
```

This produces a Sigstore attestation linked to the GitHub repo + commit + workflow. The plugin's manifest points at the attestation URL via `provenance`.

On install, `lib/plugins/safety.ts` uses the `sigstore` npm package to verify:

```typescript
import { verify } from 'sigstore';

async function verifyProvenance(manifest: PluginManifest, tarball: Buffer) {
  if (!manifest.provenance) return { verified: false, reason: 'no-provenance' };
  const bundle = await fetch(manifest.provenance).then((r) => r.json());
  try {
    verify(bundle, tarball, {
      certificateIssuer: 'https://token.actions.githubusercontent.com',
    });
    return { verified: true };
  } catch (err) {
    return { verified: false, reason: (err as Error).message };
  }
}
```

### UI treatment

- Verified → green "Signed by GitHub Actions, built from [repo@commit]" badge
- Unverified but has `provenance` field → red "Signature invalid — do not install" (block)
- No `provenance` field → yellow "Unsigned — installs at your own risk" (warn, don't block)

### V2 stance

Once 10+ plugins ship with provenance consistently, promote signing to required for new plugins in v2. Grandfather existing unsigned plugins.

## Registry CI (`.github/workflows/validate-plugin-registry.yml`)

Runs on every PR that touches `public/plugin-registry/`:

1. Install deps (`pnpm install`)
2. Build `@kairos/plugin-validator` (reads from `packages/plugin-validator/`)
3. For each changed manifest in `public/plugin-registry/manifests/`:
   - Run `npx @kairos/plugin-validator path/to/manifest.json`
   - Fail on error, continue on warning
4. For each HTTP plugin, probe the endpoint's `/manifest` and assert the response matches the registry manifest (modulo timestamps)
5. For each signed plugin, verify the Sigstore bundle
6. Regenerate `public/plugin-registry/index.json` and assert no untracked diff (if there is a diff, the PR has to include the regenerated index — this prevents CI writing to the repo)
7. On merge to `main`: nothing extra — the static file is already in place and served immediately

`app/(app)/settings/marketplace/page.tsx` already has a Plugins tab that currently only toggles bundled plugins. Extend it with the same shape as the Themes tab:

- Search bar + filter by `handlesInputTypes`
- Grid of cards, each showing: icon, name, author, description, badges (signed/unsigned, distribution type)
- "Install" button on each card → install flow above
- "Installed" badge + "Update" button if a newer version exists
- "Configure" button for installed plugins → deep-links to the plugin's settings page (already exists for bundled, needs a generic settings page that renders the plugin's `configSchema` into a form)
- "Remove" button for installed plugins — confirms before deletion, then calls `DELETE /api/plugins/[installId]`

## What does NOT ship in phase 4b

- Paid plugins, revenue split, Stripe integration for plugins
- Plugin ratings or reviews — just install counts (anonymous, hosted-only)
- Automatic updates — user must click "Update"
- Multi-channel versioning (stable/beta/nightly) — semver only, one channel
- Plugin dependencies on other plugins
- WebAssembly or V8-isolate sandboxing for bundled plugins (HTTP plugins are isolated by virtue of running in the author's process)
- Private plugin registries or enterprise plugin stores
